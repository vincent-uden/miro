# Contributing to `miro-pdf`

Thank you for considering contributing to `miro-pdf`!

## I'd like to contribute

The flow of contribution **always** starts with an issue. This is to save time
for both you and me. You don't want to be spending time writing code for a
something I won't merge into the project and I don't want to spend my time
reviewing pull requests I don't understand the intentions and nuances of.

Fortunately, anyone can 
[open an issue](https://github.com/vincent-uden/miro/issues) at any time.

> [!WARNING]
>
> If you for any reason open a pull request that is **not** in response to an
> issue I will close your PR without further discussion.

*Always check if there is an existing issue discussing your desired point of
contribution. If there is, add to the discussion in that issue. Don't open a
new one.*

## I have a bug! (or something isn't working right)

Open an issue. Describe your system:

- What operating system are you using? (If on Linux, what distro and WM/DE?)
- What version of `miro-pdf` are you using?
- Any other information that might be important when investigating the bug

If the issue is related to keybindings or otherwise controlling the program
please include the relevant lines of (or the entirety of) your config file. If
you don't know what that is, it is probably located at
`~/.config/miro-pdf/miro.conf` or at `%USERPROFILE%\.config\miro.conf`.

Describe the bug:

- What did you do to cause the bug to happen?
- What happened?
- How does that differ from what you expected to happen?

## I have an idea for a feature

Open an issue.

Describe the issue:

- What would you like the program to be able to do, that it currently can't?
- Why do you want the program to have this new feature?

## I've implemented a feature

*Is there an open issue referencing your feature?*

- **Yes**: Open a pull request!
- **No**: Go back to square one and open an issue.
